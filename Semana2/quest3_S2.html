Plan de pruebas

Objetivos del Plan de Pruebas

  • Verificar la correcta funcionalidad de cada endpoint en el contexto de las entidades relacionadas: Capacitación, Tritiano, Postulante, Cliente, Célula, Rol, Proceso, Entrevista, Pregunta, Respuesta, Estadísticas, Etapa, Rol de Usuario y Usuario.
  • Garantizar que cada acción CRUD y búsqueda específica (por ID, filtros de búsqueda) devuelva el resultado esperado, que el sistema gestione errores de manera adecuada y que las búsquedas y filtros funcionen correctamente.

    • Tipos de Pruebas

    • Pruebas Unitarias: Pruebas a nivel de código para validar las funciones individuales que gestionan cada endpoint
    • Pruebas de Integración: Validación de la interacción entre módulos, asegurando que los endpoints devuelvan la información correcta cuando interactúan entre sí .
    • Pruebas de Sistema: Verificación del flujo del sistema para asegurar que todas las funcionalidades trabajen correctamente.
    • Pruebas de Aceptación: Asegurar que los endpoints cumplan con los requisitos funcionales y no funcionales esperados por los usuarios finales.

    Especificación de Pruebas por Endpoint

    General para CRUD:

    • Listar (GET): Verificar que se devuelvan todos los registros de la entidad solicitada, y que se muestren con la estructura y cantidad de datos esperados.
    • Crear (POST): Validar que los datos ingresados se registren correctamente en la base de datos y que los errores (campos faltantes o datos incorrectos) generen mensajes de error adecuados.
    • Editar (PUT/PATCH): Comprobar que los datos de un registro existente se actualicen correctamente y que la respuesta contenga la confirmación de éxito.
    • Eliminar (DELETE): Confirmar que el registro se elimine y no sea accesible posteriormente; asegurar que la eliminación de registros no afecte otras entidades dependientes.
    Capacitación, Tritiano, Postulante, Cliente, Célula, Rol, Proceso, Entrevista, Pregunta, Respuesta:
    • Obtener por ID (GET): Validar que el sistema devuelva el registro correcto al buscar por ID, y que responda con un error en caso de un ID inexistente.
    • Listados con filtros: Para endpoints con filtros de paginación, nombre, apellido, rol, célula, y fecha, comprobar que los filtros funcionen correctamente y que los resultados sean precisos.

    Estrategia de Ejecución de Pruebas

    Define la frecuencia y el orden de ejecución de las pruebas:

    • Pruebas Automatizadas: Automatizar pruebas de endpoints CRUD y obtención de datos por ID usando Postman. Se deben considerar pruebas de regresión cada vez que se hagan cambios en el código.
    • Pruebas Manuales: Para flujos complejos y casos de aceptación de usuarios, se realizaran pruebas manuales para asegurar que los escenarios cumplan con los requisitos.
    • Pruebas de Seguridad: Asegurar que todos los endpoints protegidos por autenticación y autorización (JWT) están correctamente implementados y que los datos sensibles no sean accesibles sin permisos.

    Criterios de Aceptación

    Define cuándo se considera que una prueba es satisfactoria:

    • Funcionalidad: Todos los endpoints CRUD y de búsquedas específicas deben responder correctamente en términos de estructura, datos y errores gestionados adecuadamente.
    • Seguridad: Las pruebas de autenticación y autorización deben asegurar que sólo los usuarios autorizados accedan a los endpoints protegidos.
    • Compatibilidad: Los endpoints deben responder correctamente en diversos entornos de prueba, como diferentes navegadores y sistemas operativos.

    Informe de Resultados y Seguimiento de Errores

    Documentar y seguir los resultados de las pruebas:

    • Documentación de Resultados: Para cada caso de prueba, se documentara si la prueba pasó o falló, y se registraran los errores de forma detallada.
    • Registro de Errores: Se utilizara una herramienta de seguimiento de incidencias (como Jira, Trello o cualquier otro gestor de bugs) para registrar, priorizar y hacer seguimiento de los errores encontrados durante las pruebas.
    • Seguimiento de Correcciones: Verificar que todos los errores reportados sean corregidos y realizar pruebas adicionales para confirmar que las correcciones no afectaron otras funcionalidades del sistema.
